\section{Kisa Ozet}
Bu dokumanin amaci, mevcut RL politikasini STM32 tarafina \textbf{dogrudan} tasimaktir.

Burada gereksiz ROS ayrintisi yoktur. Yapilacak is kisaca sudur:
\begin{enumerate}
  \item Sensorden gerekli verileri oku.
  \item 48 boyutlu observation vektorunu olustur.
  \item ONNX modeli calistir ve 12 boyutlu action al.
  \item Action'i clamp et.
  \item Hedef eklem acisini hesapla:
  \[
  q_{\mathrm{des}} = q_{\mathrm{default}} + 0.5 \cdot action_{\mathrm{clipped}}
  \]
  \item Bu \texttt{q\_des} degerini mevcut motor kontrolune ver.
\end{enumerate}

\noindent
Eger STM tarafinda zaten \texttt{TFF + PD} varsa, politika tarafindan asil gereken sey \texttt{q\_des} uretmektir.

\section{Onemli Not}
Mevcut backend'e gore politika dongusu \textbf{50 Hz}'dir.

Yani birebir ayni davranisi istiyorsan:
\begin{itemize}
  \item ONNX inference: \textbf{50 Hz}
  \item Motor kontrol dongusu: \textbf{350 Hz} olabilir
  \item 350 Hz dongude son hesaplanan \texttt{q\_des} tutulur
\end{itemize}

\noindent
Eger ONNX'i direkt 350 Hz calistirirsan, backend ile birebir ayni davranis olmayabilir.

\section{\(q_{\mathrm{default}}\) Degerleri}

\begin{lstlisting}
static const float q_default[12] = {
  0.0f,  0.3f, -1.2f,   // RF
  0.0f, -0.3f,  1.2f,   // LF
  0.0f,  0.3f, -1.2f,   // RH
  0.0f, -0.3f,  1.2f    // LH
};
\end{lstlisting}

\section{Observation Vektorunun Tam Icerigi}
Observation boyutu \texttt{48}'dir.

Sira su sekildedir:

\begin{center}
\begin{tabular}{>{\ttfamily}c l l}
\toprule

\midrule

3--5 & IMU acisal hiz & 3 \\

9--20 & eklem acilari: \(q - q_{default}\) & 12 \\
21--32 & eklem hizlari \(\dot q\) & 12 \\
33--44 & previous action & 12 \\

\bottomrule
\end{tabular}
\end{center}

\subsection*{Quaternion}
Quaternion sirasi:
\begin{lstlisting}
[w, x, y, z]
\end{lstlisting}

\subsection*{Projected Gravity}
Asagidaki gibi hesaplanir:
\begin{lstlisting}
R = quat_to_rotmat(q_imu);      // q_imu = [w, x, y, z]
g_world = [0, 0, -1];
g_proj = transpose(R) * g_world;
\end{lstlisting}

\subsection*{Local Linear Velocity Icin Pratik Fallback}
Ideal durumda observation icindeki govde lineer hizi \textbf{olculen veya kestirilen local/body frame velocity} olmalidir.

\noindent
Ama eger gercek sistemde su anda:
\begin{itemize}
  \item odom yoksa
  \item velocity estimator yoksa
  \item dogrudan body linear velocity olcumu yoksa
\end{itemize}
gecici ve pratik fallback olarak sunu kullanabilirsin:

\begin{lstlisting}
v_body_fallback = [vx_policy, vy_policy, 0.0f]
\end{lstlisting}

\noindent
Yani observation icindeki ilk 3 elemani:
\begin{lstlisting}
obs[0] = vx_policy;
obs[1] = vy_policy;
obs[2] = 0.0f;
\end{lstlisting}
seklinde doldurabilirsin.

\noindent
Bu cozum \textbf{ideal degildir}, ama sistemde hic velocity bilgisi yoksa sifirdan daha iyi bir baslangic olabilir. Politika, en azindan komut edilen yon/hiz bilgisini observation tarafinda gormus olur.

\noindent
Onerilen oncelik sirasi:
\begin{enumerate}
  \item gercek local/body linear velocity
  \item estimator cikisi
  \item fallback olarak desired velocity input
\end{enumerate}

\section{Komut Isaretleri}
Politika icine girmeden once komut su sekilde ayarlanir:

\begin{lstlisting}
vx_policy = clamp(-vx_cmd, -2.0f, 2.0f);
vy_policy = clamp(-vy_cmd, -2.0f, 2.0f);
wz_policy = clamp( wz_cmd, -2.0f, 2.0f);
\end{lstlisting}

\noindent
Yani:
\begin{itemize}
  \item \texttt{vx} ters donuyor
  \item \texttt{vy} ters donuyor
  \item \texttt{wz} ayni kaliyor
\end{itemize}

\section{Previous Action}
Observation icindeki \texttt{previous action}, bir onceki step'in \textbf{clamp edilmis} aksiyonudur.

Baslangicta:
\begin{lstlisting}
previous_action[i] = 0.0f;   // i = 0..11
\end{lstlisting}

\section{Action'dan \(q_{\mathrm{des}}\) Hesabi}
ONNX cikisi 12 boyutlu bir \texttt{action} vektorudur.

Her eleman icin:
\begin{align*}
a_{\mathrm{clipped}} &= \mathrm{clamp}(a,\,-1,\,+1) \\
q_{\mathrm{des}}[i] &= q_{\mathrm{default}}[i] + 0.5 \cdot a_{\mathrm{clipped}}
\end{align*}

\noindent
Burada ekstra rate limit yoktur cunku:
\begin{lstlisting}
actionRateLimit = 0.0
\end{lstlisting}

\subsection*{Direkt Kod}
\begin{lstlisting}
for (int i = 0; i < 12; ++i) {
  float a = action_raw[i];

  if (a >  1.0f) a =  1.0f;
  if (a < -1.0f) a = -1.0f;

  action_clipped[i] = a;
  q_des[i] = q_default[i] + 0.5f * a;
}
\end{lstlisting}

\section{Motor Tarafina Ne Verilecek?}
Eger sizde motor modeli \texttt{TFF + PD} ise, politika tarafindan giden esas bilgi:
\begin{itemize}
  \item \texttt{q\_des[12]}
\end{itemize}

\noindent
Basit kullanim:
\begin{lstlisting}
qd_des[i] = 0.0f;
tau_ff[i] = 0.0f;
\end{lstlisting}

\noindent
Sonra mevcut motor katmanin zaten sunu yapiyorsa:
\[
\tau = \tau_{\mathrm{ff}} + K_p(q_{\mathrm{des}} - q) + K_d(\dot q_{\mathrm{des}} - \dot q)
\]
bu yeterlidir.

\section{Minimal STM Akisi}
\subsection*{1. Motor dongusu: 350 Hz}
\begin{lstlisting}
read_joint_state(q, dq);
read_imu(quat_wxyz, gyro);
read_body_velocity_local(v_body);

// q_des, qd_des, tau_ff son policy step'ten geliyor
motor_control_step(q_des, qd_des, tau_ff, kp_cmd, kd_cmd);
\end{lstlisting}

\subsection*{2. Policy dongusu: 50 Hz}
\begin{lstlisting}
// 1) command duzelt
vx_policy = clamp(-vx_cmd, -2.0f, 2.0f);
vy_policy = clamp(-vy_cmd, -2.0f, 2.0f);
wz_policy = clamp( wz_cmd, -2.0f, 2.0f);

// 2) projected gravity hesapla
g_proj = projected_gravity_from_quat(quat_wxyz);

// 3) observation olustur
obs[0..2]   = v_body[0..2];
obs[3..5]   = gyro[0..2];
obs[6..8]   = g_proj[0..2];

for (i = 0; i < 12; ++i) {
  obs[9  + i] = q[i]  - q_default[i];
  obs[21 + i] = dq[i];
  obs[33 + i] = previous_action[i];
}

obs[45] = vx_policy;
obs[46] = vy_policy;
obs[47] = wz_policy;

// 4) ONNX
onnx_run(obs, action_raw);

// 5) action -> q_des
for (i = 0; i < 12; ++i) {
  float a = action_raw[i];
  if (a >  1.0f) a =  1.0f;
  if (a < -1.0f) a = -1.0f;

  previous_action[i] = a;
  q_des[i] = q_default[i] + 0.5f * a;
  qd_des[i] = 0.0f;
  tau_ff[i] = 0.0f;
}
\end{lstlisting}

\noindent
Eger \texttt{v\_body} mevcut degilse, bu satiri soyle degistirebilirsin:

\begin{lstlisting}
obs[0] = vx_policy;
obs[1] = vy_policy;
obs[2] = 0.0f;
\end{lstlisting}

\section{Bu Isin En Onemli 5 Noktasi}
\begin{enumerate}
  \item Joint order'u her yerde ayni tut.
  \item Observation'i tam 48 boyut ve tam bu sirada kur.
  \item \texttt{vx} ve \texttt{vy} isaretini terse cevir.
  \item \texttt{q\_des = q\_default + 0.5 * action} formuluyle git.
  \item Birebir ayni davranis istiyorsan policy'yi 50 Hz calistir.
\end{enumerate}

\section{Son Soz}
Arkadasin sifirdan butun backend'i yazmasina gerek yok.

STM tarafinda ihtiyaci olan seyler sadece sunlar:
\begin{itemize}
  \item ONNX model
  \item bu dokumandaki observation sirasi
  \item \texttt{q\_default}
  \item action clamp ve \texttt{q\_des} hesabi
  \item 50 Hz policy dongusu
\end{itemize}

\noindent
Bunlari uygularsa mevcut politikanin STM32 tarafinda calismasi icin gerekli temel bilgi tamamlanmis olur.

\section{Odom Yoksa: Teacher-Student Yaklasimi}
Eger gercek robot tarafinda odom veya acik bir base velocity estimator yoksa, en temiz cozumlerden biri \textbf{teacher-student} yaklasimidir.

\subsection*{Temel Fikir}
\begin{itemize}
  \item \textbf{Teacher policy}: sim icinde privileged bilgi kullanir.
  \item \textbf{Student policy}: sadece gercekte elde edilebilecek sensorleri kullanir.
\end{itemize}

\noindent
Teacher sunlari gorebilir:
\begin{itemize}
  \item gercek base linear velocity
  \item gercek contact bilgisi
  \item tam state
\end{itemize}

\noindent
Student ise sadece sunlari alir:
\begin{itemize}
  \item IMU quaternion veya projected gravity
  \item IMU angular velocity
  \item IMU linear acceleration
  \item joint position
  \item joint velocity
  \item previous action
  \item command
\end{itemize}

\section{Neden Daha Iyi Olabilir?}
Bu yaklasim, politikaya ``ham sensorlerden kendi icinde state tahmini yap'' deme fikrinden genelde daha saglamdir.

Avantajlari:
\begin{itemize}
  \item egitim daha kolay olur
  \item teacher davranisi student'a net hedef verir
  \item odom olmadan deploy etmek kolaylasir
  \item ayri bir estimator gerekmeyebilir
\end{itemize}

\section{Onemli Nokta: History Gerekir}
Sadece tek bir IMU ornegi vermek genelde yeterli olmaz.

\noindent
Student policy'nin daha saglam olmasi icin genelde sunlardan biri kullanilir:
\begin{itemize}
  \item son \(N\) observation'i stack etmek
  \item recurrent policy kullanmak (GRU/LSTM)
  \item kisa zaman gecmisi ile implicit velocity tahmini yaptirmak
\end{itemize}

\section{Pratik Oneri}
Eger uzun vadede odomsuz donanim hedefleniyorsa, en mantikli yol:
\begin{enumerate}
  \item mevcut velocity-bilgili policy ile iyi bir teacher egitmek
  \item student policy'yi sadece deploy edilebilir sensorlerle teacher'a imitation ettirmek
  \item donanimda student policy'yi kullanmak
\end{enumerate}

\noindent
Kisaca:
\begin{itemize}
  \item \textbf{sim ve arastirma icin}: privileged velocity kullanan teacher mantikli
  \item \textbf{gercek donanim ve odomsuz deploy icin}: teacher-student genelde daha iyi
\end{itemize}
